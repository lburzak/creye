# ADR-019 Combined Diff Approval

## Problem & Context
- REQUIREMENTS requires the combined diff to let the reviewer approve directly from the diff: a file-level approval control per file, a gutter icon toggling approval for classes/symbols (filled green circle approved, unfilled green circle unapproved, click toggles), and highlighting of approved code blocks.
- The combined diff is already the host for two decoration concerns: gutter icons (ADR-015) and content range highlighting (ADR-016). Approval affordances are a third use of the same surface, not a new diff mechanism.
- The approval state, its descendant-validation semantics, and its persistence are owned by ADR-018; this record decides only how approval is shown and toggled inside the diff.
- The diff lives in the ide layer (ADR-001, ADR-015 constraint 3); it must mutate approval through callbacks into the ADR-018 model, never by reaching into persistence or the Compose rendering surface.

## Constraints
1. Approval affordances MUST appear only in the plugin's own combined diff view, reusing the ADR-015 processor-scoped editor walk; they MUST NOT decorate diffs the plugin did not open.
2. A file-level approval control MUST be displayed for each file block.
3. A gutter icon MUST indicate the approval status of classes and symbols: a filled green circle when validated, an unfilled green circle when not, and clicking it MUST toggle approval.
4. Approved code blocks MUST be highlighted within the diff.
5. Toggling approval MUST mutate the ADR-018 model through ide-layer callbacks; it MUST NOT write persistence directly nor reach into the rendering surface (ADR-001, ADR-015 constraint 3).
6. An affordance MUST degrade safely when a block exposes no usable editor (ADR-015 constraint 4).

## Decision
- The class/symbol gutter toggle MUST be a **contribution to the ADR-015 gutter-icon facility**, not a separate decoration path: ADR-019 supplies the correlation node, a **range anchor** from the node's ADR-004 changed-symbol range, the icon, the tooltip, and the click `AnAction`; ADR-015 owns the scoped editor walk, block-to-node correlation, placement, dedup, stacking, and disposal. The contributed icon MUST be resolved from the node's ADR-018 **effective validation** — filled green circle when validated, unfilled green circle otherwise — and the click action MUST invoke the node-scoped approval-toggle callback (constraints 3, 5).
- The file-level control MUST be the file node's approval toggle, contributed to the same ADR-015 facility as a **block-anchored** icon at the block's first line. Toggling it MUST ripple per ADR-018: for a file with declaration children it writes/removes an entry for every changed leaf in the file; for a leaf file node (ADR-017) it writes/removes that file's own entry (constraint 2).
- Approved-block highlighting MUST be built on the ADR-016 range-highlighting mechanism and MUST be applied at **leaf granularity**: one `RangeHighlighter` per validated leaf node over its ADR-004 changed-symbol range (the same source ADR-016 drives content highlighting from), present iff that leaf carries an entry per ADR-018. A container MUST NOT get its own block-spanning highlight; an "approved file" is shown by its leaves' ranges all being highlighted, not by a whole-block fill. Highlights obey ADR-016's editor-local markup, side resolution, and above-diff-background layer rules (constraints 1, 4).
- The class/file gutter circle MUST be binary — filled green iff the node is fully effectively validated (all its changed leaves validated), unfilled green otherwise, including the *partial* case. The diff MUST NOT add a third (partial) gutter state. Partial validation is conveyed instead by the per-leaf circles the diff always renders alongside the container circle: a partly-approved class shows an unfilled class circle with a mix of filled and unfilled symbol circles beneath it. This is the diff's honest rendering of ADR-023 full/partial/none without a partial glyph (constraint 3).
- The toggle's write target MUST follow ADR-018's ripple: clicking a leaf node adds/removes that leaf's own entry; clicking a container (file/class) adds the rippled entry set for its changed leaves if not fully validated, and removes that set if it is. The displayed icon and highlight MUST reflect effective validation per ADR-018 — a leaf by its own entry, a container by all-its-changed-leaves-validated — so every node a user clicks is individually toggleable: a symbol under an approved file has its own entry and can be un-approved on its own, dropping the file to partial validation.
- The per-node line ranges this record's gutter anchors and ADR-016 highlights depend on MUST reach the ide layer by being carried on the domain change set: the current-side range MUST be retained on `ChangedDeclaration` (it is computed during ADR-004 detection but currently dropped at the domain boundary) and the ide diff MUST read the change set (`ChangedSymbols`, keyed by `NodePath`) to resolve a node's range. Ranges MUST NOT be pushed into the rendering projection (`VisibleNode`); the Compose surface stays range-free per ADR-011, and only the ide layer — which may see analysis data — consumes them.
- All approval decorations MUST be rebuilt from current ADR-018 state when the diff opens and after each toggle — a full rebuild over the open diff's change set, keeping decorations a pure function of model state with no incremental-update bookkeeping — and MUST be released with the diff dialog's `Disposable` alongside the ADR-015/ADR-016 decorations (constraint 6 degradation included). Scoping the rebuild to the toggled subtree and its ancestor chain is a deliberate non-goal until profiling shows the full rebuild is too costly.

## Rationale
- Contributing the gutter toggle to the ADR-015 facility and the block highlight to ADR-016 keeps approval a thin consumer of mechanisms those records own: ADR-019 supplies icon, anchor, and action and nothing more, so it inherits ADR-015's scoping, correlation, deterministic gutter stacking, and disposal instead of re-deriving them or introducing a global `DiffExtension` whose project-wide blast radius ADR-015 rejected.
- Resolving icon color and highlight presence from ADR-018 effective validation, and delegating the ripple to the model on toggle, keeps the diff a thin view: the diff names the clicked node and asks the model to toggle it; the container-to-leaf ripple and the all-leaves-validated derivation live in ADR-018/ADR-023, not in the diff. A class whose symbols are only partially validated is therefore not effectively validated and shows an unfilled circle, with no partial-state logic in the diff itself.
- Expressing toggles as callbacks into ADR-018 preserves the ADR-001/ADR-011 boundary — the diff never touches persistence and the Compose surface never sees editors or VCS state.
- A binary class circle is consistent with the graph's ternary marker (ADR-020) rather than in conflict with it: the two views agree because they derive from one ADR-018 model, not because they use one glyph vocabulary. The diff can afford binary because it always shows the leaf circles that spell out the partial state; the graph needs ternary precisely because a collapsed node (ADR-008) hides those leaves and must aggregate. Forcing a third gutter glyph onto the diff would add a state REQUIREMENTS does not ask for and that the leaf circles already express.

## Notes
- Open, shared with ADR-015/ADR-016's deferred decoration vocabulary: the exact rendering of the filled/unfilled green circle and of the approved-block highlight. The file control's placement is decided above — a gutter icon at the block's first line, reusing the ADR-015 mechanism rather than adding a block-header surface — not deferred.
- Under ADR-018's rippled model, every symbol under an approved file carries its own leaf entry and can be un-approved individually from the diff; doing so drops the file to partial validation (its gutter circle goes unfilled). There is no revoke-via-ancestor indirection and no silent toggle.
- Gutter column sharing (an earlier open question) is resolved by ADR-015 becoming the general gutter-icon facility: it owns deterministic stacking of contributions from multiple features, so the approval circle's coexistence with any nav/marker icon is ADR-015's rule, not this record's.
- Highlight-overlap-between-levels (an earlier open question) is resolved by the leaf-granularity highlight decision above: there is no container-level block highlight to overlap a leaf range, so nested highlights cannot stack.
- ADR-004 range exposure (an earlier open question) is resolved by inspection of the code: detection computes per-declaration ranges (`ChangedSymbolDetector` `DeclNode.fullRange`/`ownRanges`) but drops them at the domain boundary (`ChangedDeclaration`), so the ide diff (`NodeDiffPresenter`) currently sees only `NodePath` and full-file Git changes. The Decision above closes this by retaining the current-side range on `ChangedDeclaration` and having the ide read the change set; this imposes a downstream-contract extension on ADR-004's output, noted there.
- The combined diff view itself is still not captured by its own ADR (noted in ADR-015/ADR-016); if recorded, this approval record would reference it as the host.
