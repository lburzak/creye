# ADR-018 Approval Model

## Problem & Context
- A reviewer needs to mark parts of the change set as reviewed/approved. Approval is review-progress state laid over the changed-code graph (ADR-007) and the combined diff, not an analysis output. Approval asserts that **the change is acceptable** — a judgment about the before→after transition the reviewer saw, not about the resulting code in isolation; ADR-022 fingerprints both diff sides accordingly.
- Approval applies to graph nodes at any containment level — module, package, file, class, symbol — so the model must address all ADR-005 structural node kinds, not just files.
- REQUIREMENTS requires approval status to be persisted and restored across sessions, and to be cleared whenever the diff changes.
- Approving a container (e.g. a module) MUST mean each of its changed descendants is independently approved, so that a later content change invalidates only the descendant that changed, not the whole container approval. This requires per-descendant approval state: a single container entry cannot record what each descendant looked like at approval time, so it cannot invalidate granularly. The cost — approving a container is no longer a single entry, and undoing it is no longer a single removal — is accepted in exchange for granular invalidation.
- The literal "clear approval whenever the diff changes" is coarse: re-running analysis after an unrelated edit would wipe approvals the reviewer already completed. The intent is to invalidate approvals whose underlying content changed, granularly.
- ADR-005 identity is deliberately run-local: it "MUST NOT persist or be compared across runs." Approval must persist and be matched against a future run's nodes, which is in direct tension with that rule and must be resolved here rather than by quietly persisting a `NodePath` handle.

## Constraints
1. Approval state MUST be representable for any ADR-007 structural node kind (module, package, file, class, symbol); external nodes (ADR-007) are out of scope.
2. Approval entries MUST be stored only on leaf (own-content) nodes — changed symbols and leaf file nodes. Approving a container MUST ripple to an entry for every changed leaf in its subtree; a container itself MUST NOT carry an entry.
3. A leaf node MUST be derivable as validated iff it carries its own entry. A container node MUST be derivable as validated iff every changed leaf in its subtree is validated. Validation MUST NOT be stored for any node, only derived from the leaf entries.
4. Approval state MUST persist across sessions and MUST be restorable when the graph or combined diff reopens.
5. Approval persistence MUST NOT store the run-local ADR-005 `NodePath` handle; it MUST key on the persistable segment-value serialization ADR-005 permits for cross-run consumers.
6. An approval entry MUST be invalidated when its node's diffed content changes between runs, and MUST be retained when that content is unchanged.
7. Undo of an approval action MUST restore the exact set of leaf entries the action wrote or removed: a single leaf approval is one entry, a container approval is the rippled leaf set, undone atomically.

## Decision
- Approval state MUST be a map from a **persistable leaf-node key** to an approval record. Entries exist only for leaf (own-content) nodes; a single user action may write or remove many entries (one per changed leaf in the acted-on node's subtree). The map MUST live in the domain layer (ADR-001) and be exposed to the ide and rendering layers as state plus mutation callbacks; the rendering surface MUST NOT read or write the persistence store directly (ADR-001, ADR-011).
- The persistable node key MUST be a deterministic serialization of the ADR-005 structural-path segment **values** (the kind-tagged segment list down to the node), NOT the interned run-local handle. Matching a persisted approval to a live node within a run MUST be done by comparing this serialized form against the serialization of the freshly minted `NodePath`, never by persisting or restoring a handle (constraint 5).
- Approving a leaf node MUST write one entry for that leaf's key. Approving a container MUST write an entry for every changed leaf in its subtree (the ripple), keyed by each leaf; the container itself receives no entry (constraint 2). The **effective validation** of a node MUST be derived at read time: a leaf is validated iff it has its own entry; a container is validated iff every changed leaf in its subtree is validated (constraint 3). This makes a single descendant change invalidate only that descendant's entry, leaving its siblings approved — the granular invalidation REQUIREMENTS asks for. Undo restores the exact entry set the action wrote (constraint 7).
- Each approval entry MUST bind a **content fingerprint** of the approved node's diffed content captured at approval time (derived from the node's ADR-004 change kind and changed-content range). On every analysis run, an entry whose recomputed fingerprint differs from the stored one MUST be dropped, and an entry whose fingerprint matches MUST be kept (constraint 6). This fingerprint-scoped invalidation replaces the literal blanket "clear on every diff change": an unchanged node stays approved, a changed node loses approval.
- The persisted approval state MUST carry a **fingerprint-scheme version** identifying the ADR-022 fingerprint definition (algorithm, normalization, both-side input, change-kind folding) that produced its stored fingerprints. On load, entries whose stored scheme version does not match the running version MUST be dropped as uncomparable — a defined, logged invalidation, never a silent comparison of fingerprints minted under different schemes. The MVP performs no cross-scheme migration; a scheme change means re-review, which is acceptable and rare. This makes a future normalization or algorithm change (ADR-022's deferred lever) detectable instead of a silent mass mismatch.
- Approval state MUST be persisted in per-project plugin state (`PersistentStateComponent`), consistent with ADR-011's per-project persisted configuration, and restored on graph/diff open (constraint 4).
- Approval entries MUST only be minted for structural nodes (constraint 1); a request to approve an external node MUST be ignored, not persisted.

## Rationale
- Storing entries only at leaves and rippling container approvals down to them is what makes granular invalidation honest: each leaf carries its own ADR-022 fingerprint, so a change to one leaf drops only that leaf's entry and its siblings stay approved. A single container entry could not do this — it would need to remember every descendant's content at approval time, which is the per-descendant state we are choosing to store outright rather than fake. The cost is paid in reversal: a container approval is a set of leaf entries, so revoking it ripples the inverse over the current leaf set (ADR-025) rather than clearing one entry.
- Deriving container validation upward (validated iff all changed leaves are) rather than storing it means a leaf added under an already-approved container is correctly unvalidated until reviewed: the container simply stops being fully validated, with no stale container entry asserting otherwise.
- Binding each entry to a content fingerprint turns the coarse "clear on diff change" requirement into granular invalidation: re-analysis after an unrelated edit keeps approvals whose nodes are byte-for-byte the same and drops only those whose content moved, which is the behavior the reviewer actually wants.
- Serializing ADR-005 segment values rather than persisting the handle keeps the in-memory `NodePath` handle run-local while still giving approval a stable cross-run key, because the segment text is deterministic for unchanged code. ADR-005 was amended to permit exactly this segment-value serialization for cross-run consumers, so the approval key rests on a sanctioned affordance rather than a contradiction.
- Per-project `PersistentStateComponent` reuses the persistence seam ADR-011 already established for the comparison branch, so approval does not introduce a parallel storage mechanism.

## Notes
- The content-fingerprint definition and its formatting-sensitivity are owned by ADR-022, which applies per leaf entry. With container approvals rippled to leaves, there is no container entry and therefore no aggregate container fingerprint; ADR-023 now owns how container validation completeness is *derived* from leaf entries, not an aggregate fingerprint. ADR-022 decides the leaf fingerprint is a hash of the node's verbatim diffed content on **both sides** (before and after) plus its ADR-004 change kind, with no normalization for the MVP; this is what makes approval bind to the change rather than the resulting code, and makes deletions approvable.
- Missing-node invalidation and rename/move matching are owned by ADR-024 (staged key match, then move/rename-mapping migration, else clean drop); container-entry invalidation by ADR-023.
- Validation derivation runs over the live ADR-007 materialized graph: a leaf's key is serialized and checked against the map; a container is walked *downward* over its materialized changed leaves, each leaf key serialized and checked, and the container is validated iff all are present. No `NodePath` handle is persisted or restored; leaf-entry retention is decided by ADR-022.
- The persistable key rests on ADR-005's amended rule, which now explicitly permits persisting and cross-run comparing a serialization of the segment values; only the in-memory handle stays run-local.
- A leaf entry is the sole carrier of approval state: a leaf is either deliberately approved (own entry) or unapproved. Because container approvals ripple to leaf entries, every leaf under an approved container has its own entry and CAN be individually un-approved — toggling it off removes just that leaf's entry, dropping the container to partial validation. There are no negative or override entries; revocation is always entry removal. The UI records (ADR-019, ADR-020) toggle the rippled leaf set for a container and the single entry for a leaf.
- Map size is bounded by the count of changed leaf nodes in the diff, not the whole tree: rippling a container approval writes one entry per *changed* leaf under it, so the persisted map scales with diff size, not codebase size.
- This record owns the model and persistence only. How approval is triggered and shown is split across ADR-019 (combined diff) and ADR-020 (graph node), both of which derive effective validation from this model.
